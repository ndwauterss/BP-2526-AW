\chapter{\IfLanguageName{dutch}{Proof-of-Concept}{Proof-of-Concept}}%
\label{ch:proof-of-concept}

Dit hoofdstuk beschrijft de realisatie van de Proof-of-Concept (PoC). Er wordt gestart met een technische analyse van de huidige toepassing binnen het probleemdomein, gevolgd door een verantwoording van de migratie naar een modern, container-native framework. Vervolgens wordt besproken hoe de testomgevingen in Azure Container Apps en Google Cloud Run opgezet zijn. Bij een PoC horen ook steeds technische uitdagingen en limitaties die aangekaart moeten worden, aangezien ze aanzetten tot bepaalde keuzes. Na dit cruciaal onderdeel volgt een bespreking van de strategieën die toegepast zijn om de meetdata te borgen. Tot slot worden de tools, uitvoeringsprotocol en testcases gedefinieerd om de lezer een volledig inzicht te geven in hoe de oplossingen zijn opgezet en gevalideerd.

\section{Technische Analyse van de Bestaande Toepassing}

De huidige toepassing is tweeledig. Enerzijds is er een HttpTrigger die de synchrone verwerking voor zich neemt. Anderzijds bevat het algoritme een QueueTrigger die de asynchrone verwerking van de zware taak uitvoert. De reden van deze opzet is om de gebruiker zo snel mogelijk een antwoord te kunnen geven dat de generatie van de dataset begonnen is. Op deze manier moet er niet gewacht worden op de volledige verwerking van de set, wat meerdere minuten in beslag neemt. De asynchrone verwerking update de status van de dataset op bepaalde momenten in de verwerking. Deze status wordt om de 5 seconden opgevraagd in het scherm waar het proces geactiveerd werd en maakt het mogelijk voor de gebruiker om tussendoor andere acties te doen in Agromanager. 

\subsection{HttpTrigger} 

De HttpTrigger is een Azure function die een POST endpoint ter beschikking stelt voor Agromanager. Dit kan aangeroepen worden vanuit Agromanager en ontvangt bepaalde gegevens die noodzakelijk zijn voor de opstart van het proces. Deze gegevens komen binnen via de Body van een HttpRequest object. De Body wordt vervolgens omgezet naar een string in JSON-formaat door middel van een StreamReader en in de volgende stap gedeserialiseerd naar een DatasetGeneratorRequest object. Dit object bevat volgende eigenschappen:
\begin{description}
    \item [GpsDatasetID:] 
    ID van de dataset, nodig om op te halen welke teelten hieraan gekoppeld zijn en welke bestanden met meetdata hier bijhoren.
    \item[GpsDatasetSource:] 
    Een enum-waarde die later gebruikt zal worden om te bepalen welk type processor er nodig is bij het genereren. Bv.: Shapefile, CSV, etc.
    \item[AgroID:] 
    ID van de Agromanager-omgeving. Dit wordt gebruikt om te bepalen welke database er nodig is voor het ophalen en wegschrijven van data.
    \item[CompanyName:] 
    Bedrijfsnaam van de omgeving. Dit heeft dezelfde functie als een AgroID.
\end{description}

Het verkregen object wordt in de volgende stap naar een Azure Storage Queue gestuurd, waar het in een wachtrij terecht komt om verwerkt te worden. Tot slot wordt een OkObjectResult teruggeven naar de aanvrager wat resulteert in een Status 200 OK.

\subsection{QueueTrigger}

\subsubsection*{Highlevel}

Wanneer het bericht in de wachtrij opgepikt wordt om te verwerken, wordt de QueueTrigger aangeroepen. Dit bevat het zwaarste onderdeel van het proces en start met het bepalen van welke FileProcessor er nodig is op basis van de waarde in de GpsDatasetSource-eigenschap van het DatasetGeneratorRequest-object. Deze keuze wordt gedaan door gebruik te maken van het Strategy Pattern. Deze FileProcessors passen elk de interface IFileProcessor toe, die een functie beschikbaar maakt die dynamisch ingevuld kan worden in de runtime van de code. Dit maakt het mogelijk om later makkelijk nieuwe FileProcessors toe te voegen zonder de bestaande code drastisch te moeten aanpassen. In de QueueTrigger wordt enkel de logica van het Strategy Pattern aangeroepen, gevolgd met de aanroep van de dynamische functie. De logica van het Strategy Pattern wordt voorafgegaan met een using-statement om de database te disposen, nadat het proces is uitgevoerd. Dit is nodig omdat de instantie van de database manueel aangemaakt wordt en dus ook manueel opgekuist moet worden. 

\subsubsection*{FileProcessor}

De effectieve verwerkingslogica van de dataset bevindt zich in de implementatieklasses van de IFileProcessor-interface. Elke FileProcessor volgt in grote lijnen dezelfde structuur, maar verschilt op het vlak van ophalen en verwerken van de inhoud van de bestanden die gelinkt zijn aan de dataset en die de boomdata bevatten. Hoe deze logica functioneel werkt werd eerder besproken in het hoofdstuk van de Requirementsanalyse (\ref{pr:flow-generatie}). 

\subsection{Computationele Karakteristieken}

Om beter te begrijpen waarom de verwerkingslogica als zwaar wordt gezien en waarom het meerdere minuten kan duren om een dataset te generen, wordt hier besproken wat de karakteristieken zijn.

\subsubsection*{Geneste Iteraties}

De code bevat diepe geneste foreach-lussen die zorgen voor een verwerkingscomplexiteit van $O(n^4)$. Bij grote datasets zorgt dit voor een exponentiële toename van de verwerkingstijd.

\begin{equation}
    O(T \cdot B \cdot L \cdot P) \approx O(n^4)
\end{equation}
\label{frm:iteraties}
waarbij:
\begin{itemize}
    \item $T$ = Aantal Teelten: loopt door alle teelten gelinkt aan de dataset.
    \item $B$ = Aantal Bestanden: loopt per teelt door alle bestanden gelinkt aan de dataset.
    \item $L$ = Aantal Punten/Lijnen (CSV-regels of Shapefile features): loopt per teelt, per bestand door alle lijnen of punten die uit het bestand komen.
    \item $P$ = Aantal Bomen in het perceel: loopt per teelt, per bestand, per lijn of punt uit bestand door alle bomen in de teelt.
\end{itemize}

Dit zal bij een gemiddelde dataset zorgen voor miljoenen iteraties. In combinatie met geografische berekeningen zorgt dit voor een constante druk op de CPU en de schaalbaarheid van de omgeving.

\subsubsection*{Geometrische en Ruimtelijke Berekeningen}

Een verwerking in de CsvProcessor duurt minder lang en zal ook minder intensief zijn voor de CPU, dan bij de ShapeFileProcessor. Dit komt doordat er bij een CSV-bestand enkel gewerkt wordt met tekstuele waarden om de lijnen of punten uit het bestand te halen. Bij een Shapefile moeten er actieve ruimtelijke transformaties uitgevoerd worden. In de code zie je per rij in het bestand dat de functie file.Reproject(...) uitgevoerd wordt. Dit komt er op neer dat er per lijn of punt in het bestand de CPU complexe goniometrische berekeningen moet uitvoeren om coördinaten om te zetten naar de WGS1984-standaard.
\\ 
\\
Dit neemt niet weg dat de CsvProcessor ook intensief kan zijn voor de CPU. Een bestand (zowel CSV als Shapefile) kan data bevatten per Boom- en/of TeeltID. In dit geval is er minder verwerkingstijd nodig, want dan kan de meetdata rechtstreeks gekoppeld worden aan de boom. De BoomID in het bestand matcht direct met de BoomID van de teelt. Echter is het ook mogelijk dat het bestand geen Boom- of TeeltID's bevat. In dit geval moet er per punt van het bestand een cirkel getrokken worden rond elke boompunt van de teelt. Vervolgens moet berekend worden op basis van latitude en longitude of het punt in het bestand binnen de cirkel van de boom ligt. Dit legt een constante druk op de CPU. Bij een Shapefile is deze druk nog iets hoger, omdat dit meestal meer datapunten bevat dan een CSV, wat voor meer berekeningen zorgt.

\subsubsection*{Geheugenintensieve taken}

Per teelt wordt een masterfile opgehaald uit Azure Blob Storage die gedeserialiseerd wordt naar een objectmodel. Dit model wordt de hele tijd in het geheugen bijgehouden en ook gemanipuleerd door waarden aan de bomen toe te voegen. Hierdoor stijgt het RAM-gebruik en krijgt de .NET Garbage Collector veel werk, wat de verwerking onderbreekt. 

\subsubsection*{Database en I/O interactie}

Per teelt, per bestand en ook per 100 regels in het bestand wordt er een statusupdate weggeschreven naar de database. Dit is noodzakelijk om de gebruiker op de hoogte te brengen van de voortgang. Echter zorgt dit ook voor wachttijden op I/O, waardoor CPU-threads langer bezet blijven. 

\subsubsection*{Data-transformatie en Opslag}

De masterfile die als objectmodel in het geheugen leeft, wordt op het einde opnieuw geserialiseerd en naar Azure Blob Storage weggeschreven. Dit bestand kan veel data bevatten met vele megabytes tot gevolg. Dit moet in één keer gebeuren, wat ook hier zorgt voor een piek in het RAM-gebruik en de hoeveelheid werk voor de Gargbage Collector.

\subsection{Technologische Stack}

\begin{description}
    \item[.NET Core 3.1 (In-Process):]
    Het framework waarop de huidige toepassing momenteel draait. Het In-Process model houdt in dat de applicatiecode in hetzelfde proces draait als de Azure Functions-runtime, waardoor resources gedeeld worden.
    \item[Azure Functions (App Service Plan):]
    De huidige toepassing wordt gehost als een Function app in Azure. De compute-omgeving is App Service Plan. Dit houdt in dat er vaste resources zijn toegewezen die gedeeld worden door alle function apps in dit plan. 
    \item[Entity Framework Core:]
    De Object-Relational Mapper (ORM) die gebruikt wordt voor de communicatie met de onderliggende SQL Databases.
    \item[SQL Database:]
    Database die gebruikt wordt om data op te halen of te persisteren, waaronder het updaten van de status.
    \item[Azure Blob Storage:]
    Centrale opslagplaats voor bronbestanden (masterfile, CSV, Shapefile) en de resultaatbestanden.
    \item[Azure Storage Queue:]
    Wachtrij in de cloud die taken in een bepaalde volgorde afhandelt. Dit triggert de QueueTrigger in de huidige toepassing.
\end{description}

Deze stack vormt de technische baseline. Beperkingen in isolatie (In-Process) en de specifieke koppeling met Azure-native componenten zoals Azure Storage Queue, vormen de aanleiding voor de migratie naar een container-native architectuur in het kader van deze Proof-of-Concept.

\section{Migratie en Modernisatie}

Om de bestaande toepassing te migreren naar de testomgevingen werd er een nieuw project opgestart. Dit project werd voorzien met componenten van de code die noodzakelijk waren om de dataset te genereren. Andere complexiteiten werden geabstraheerd.
Bij de migratie kwam naar boven dat er aanpassingen in de code nodig waren. Dit zorgde voor een uitgelezen kans om de \textit{technical debt} te verminderen. Toch is er geprobeerd om de bestaande, noodzakelijke code zo min mogelijk aan te passen om de core functionaliteit van het genereren te behouden. Dit waarborgt een eerlijke vergelijking met de huidige toepassing en de nieuwe oplossingen, omdat de business logica hetzelfde bleef. Bij het opzetten van de testomgevingen kwamen ook nog uitdagingen naar boven die invloed hebben gehad op de code en de keuze van bepaalde tools. 

\subsection{Framework-transitie: .NET 8 Isolated Worker Model}

De huidige toepassing draait op .NET Core 3.1. Deze versie is \textit{end of life} en wordt niet verder ondersteund. Anderzijds is deze versie ook voorzien van het In-Process model wat betekent dat de runtime en de worker in hetzelfde proces draaien. Dit zijn kenmerken die niet ideaal zijn voor een toepassing die moet draaien in een container. Hierdoor werd er gekozen om de applicatie te migreren naar .NET 8 met het Isolated Worker model. Dit scheidt het proces van de runtime met het proces van de worker. De testomgevingen bevinden zich in 2 verschillende cloudproviders. Hierdoor is het een vereiste dat de runtime en de applicatiecode geïsoleerd worden van elkaar, omdat elke provider een andere runtime heeft. .NET 8 is ook gebouwd met de focus op container-omgevingen, waardoor het logisch is om de bestaande toepassing te migreren naar deze versie. Deze keuze heeft er voor gezorgd dat er Nuget packages voor de SDK van de worker geïnstalleerd moesten worden, alsook werd de versie van Entity Framework geüpdatet naar een 8.x versie. Bij het testen van de code na transitie naar de nieuwe versie kwam ook boven dat er \textit{Includes} ontbraken bij het ophalen van data via EF. Door lazy loading bleven bepaalde gelinkte objecten leeg, waardoor NullReferenceException gegooid werd. 

\subsection{Dependency Injection}

De kracht van het Isolated Worker model is dat er volledige controle is over het proces van de worker. Dit model komt ook standaard met \textit{Dependency Injection (DI)}. Dit houdt in dat de verantwoordelijkheid van de aanmaak en opkuis van bepaalde instanties van objecten geregeld worden door de applicatie. Dit kan bekomen worden door de objecten te registreren in de \textit{Program.cs} file. Zo is het verschil tussen de huidige en de nieuwe toepassing dat de initialisatie van de database niet meer manueel gebeurt via een connection string, maar via de DI. In de code heeft dit als gevolg dat de logica van het Strategy Pattern niet meer aangeroepen moet worden met een using-statement, omdat de instantie van de database automatisch opgekuist wordt nadat het proces gedaan is.

\subsection{Platformonafhankelijke Messaging}
\label{ch:messaging-queue}

Na het opzetten van de testomgevingen en het uitvoeren van de eerste testen, viel op dat de instanties van de container bij Azure Container Apps niet schaalden zoals gehoopt. Hierbij zijn meerdere pogingen gedaan om dit in orde te krijgen, waaronder: configuratie van de container aanpassen en host.json aanpassen in code. Na verder onderzoek bleek dat Azure Storage Queue niet ideaal was om dit te laten schalen zoals gewenst, omdat het een eigen manier van schalen hanteert. Hierna is besloten om dit te vervangen door Azure Service Bus, een dedicated message broker, die zorgt voor een agressievere schaling via KEDA. Om dit te bekomen in de code moest er een nuget package geïnstalleerd worden en moest de QueueTrigger aangepast worden naar een ServiceBusTrigger van de desbetreffende package. Dit gaf wel het gewenste effect. Vervolgens bij het testen van de omgeving in Google Cloud Run werd terug duidelijk dat het schalen niet verliep zoals gewenst. Dit komt omdat Google Cloud Run geen manier heeft om te kijken naar de Queue in Azure Service Bus om te bepalen of er bijgeschaald moet worden in tegenstelling zoals bij ACA met KEDA. Hierdoor werd er beslist Pub/Sub te gebruiken bij Google Cloud. Dit is de tegenhanger van Azure Service Bus in Azure en werkt op een gelijkaardige manier. Om dit te gebruiken werd in de code 3 zaken aangepast:

\begin{description}
    \item[Bericht in queue zetten:] Er werd een MessageHandler voorzien in de code die op basis van een environment variable beslist welke handler nodig is en zo het bericht in de juiste queue zet. Ook hier werd opnieuw het Strategy Pattern gebruikt. Welke instantie nodig is, wordt beslist bij de DI registratie in Program.cs. Deze handler instantie wordt beschikbaar gemaakt via de constructor en werd toegepast in de HttpTrigger.
    \item[POST endpoint specifiek voor de Pub/Sub in Google Cloud:] In tegenstelling tot Azure Service Bus werkt Pub/Sub niet op basis van een wachtrij verwerking, waarbij een QueueTrigger aangeroepen wordt. Dit werkt enkel met POST endpoints om de uiteindelijke verwerking te kunnen uitvoeren. Daarom werd beslist om de code uit te breiden met een nodige POST endpoint die de genereerlogica uitvoert.  
    \item[Abstractielaag genereerlogica:] Om dezelfde genereerlogica toe te passen voor de QueueTrigger en de POST endpoint voor Pub/Sub werd er een abstractielaag voorzien tussen de triggers en de verwerkingslogica. Dit werd in een DataProcessor object weggestoken die via DI ter beschikking gesteld wordt voor de triggers. Zo is er geen duplicatie van code nodig om exact hetzelfde proces uit te voeren in de verschillende providers.
\end{description}

\subsection{Optimalisatie Legacy Code}

Bij de eerste testen kwam een probleem naar boven met de \textit{AzureHelper}. Dit is legacy code die alle functionaliteit voorziet voor de communicatie met Azure Blob Storage. Het probleem deed zich specifiek voor bij het verkrijgen van de \textit{WriterStream} om het resultaat weg te schrijven. Alvorens deze stream verkregen werd, werd er eerst een \textit{SetHttpHeaders} uitgevoerd via de Azure API om de header op de blob te zetten. Aangezien een parallel proces dit ondertussen ook deed, of naar ditzelfde bestand aan het wegschrijven was, veranderde de status tussen de \textit{header-call} en de \textit{write-call} met een HTTP-statuscode 412 als gevolg. De Azure SDK ziet dit als een onherstelbare fout, waardoor de verwerking stopte.
\\ \\
Om dit op te lossen werd de code aangepast zodat er geen aparte call meer nodig is voor het zetten van de header. Er is namelijk een manier om de headers direct mee te geven bij het wegschrijven van het bestand. Dit voorkomt dat er een tussenstatus ontstaat tussen de \textit{header-call} en de \textit{write-call}. Bij het wegschrijven van het bestand wordt nu ook een \textit{overwrite}-flag meegegeven die aan de Azure API aangeeft dat het bestand overschreven mag worden. Door bij de \textit{write-call} zowel het bestand weg te schrijven, de headers te zetten als aan te geven dat overschrijven is toegestaan in één enkele call, werd het probleem opgelost.
\\ \\
In de huidige toepassing in het probleemdomein vindt dit probleem vrijwel niet plaats, omdat de kans dat een dataset gelijktijdig gegenereerd wordt voor eenzelfde teelt in dezelfde omgeving heel klein is, dit komt in de praktijk bijna niet voor. Verder is dit ook een optimalisatie, aangezien het aantal API-calls is gereduceerd van twee naar één.

\subsection{Data- en Opslaglaag}

\subsubsection*{Azure SQL Database}

Een belangrijk onderdeel voor de testomgeving is een database om data persistent te maken over meerdere omgevingen en testruns heen. Aangezien de bestaande toepassing in het probleemdomein ook gebruik maakt van een Azure SQL Database, was het een logische keuze om dit ook voor dit onderzoek te gebruiken.

\begin{description}
    \item[Provisionering en Resource Group:] 
    Er werd een nieuwe resource group aangemaakt binnen de subscription van Azure die gebruikt werd voor alle resources binnen Azure. De database werd ook toegewezen aan deze resource group. Als regio werd er gekozen voor \textit{West Europe}, omdat dit geografisch het dichtste aansluit met Agromanager. Dit voorkomt netwerkvertragingen.
    \item[Database Tier:]
    Er werd gekozen voor een textit{free database offer} binnen Azure. Dit houdt in dat de eerste 100.000 vCore seconden en 32GB geheugen gratis zijn per maand. Verder werd gekozen om tegen vergoeding de database verder te gebruiken, indien de limiet van de gratis aanbieding bereikt is. Dit waarborgt dat de database beschikbaar blijft. Deze aanbieding was enkel geldig voor een serverless model die gebruik maakt van vCores. Dit heeft als nadeel dat de database inactief wordt na 1 uur van inactiviteit wat voor een cold start zorgt. Deze cold start kan omzeild worden en zal later aangekaart worden. Verder werd er gekozen voor een maximum van 2 vCores en een minimum van 0.5 vCores wat resulteert in een minimum werkgeheugen van 2.05 GB en een maximum verwerkingsgeheugen van 6GB. De maximale opslag van de gegevens werd op 1GB gezet, omdat dit ruim voldoende is voor de data die aanwezig zal zijn in de database. Anderzijds drukt het ook de kosten, indien de gratis limiet is overschreden. Tot slot werd de database niet aangeduid als \textit{zone redundant} en wel als \textit{Locally-redundant backup storage}.
    \item[Authenticatie:]
    Er werd gekozen om de authenticatie van de database via SQL Server Authentication (gebruikersnaam en wachtwoord) te regelen. Dit was voornamelijk noodzakelijk voor Google Cloud Run, omdat dit niet zo eenvoudig overweg kan met Azure AD Managed Identities. Ook is het makkelijker om via een universele connection string de database connectie te regelen via een Environment Variable voor beide omgevingen. 
    \item[Beveiliging:]
    Om de database via Microsoft SQL Server Management Studio te kunnen raadplegen, was het nodig om het IP-adres van de client (laptop) toe te voegen aan de firewall-regels van SQL Server waarop de database draait. De firewall werd ook zo geconfigureerd dat Azure services toegang kregen, zonder expliciete regels te moeten definiëren. Voor Google Cloud Run moest er wel een work-around voorzien worden, omdat dit geen vast IP-adres toegewezen krijgt. Alvorens de testruns voor Google Cloud Run uit te voeren, werd eerst een extra firewall-regel toegevoegd die de deur volledig openzet. Zo werd de start IPv4 adres op 0.0.0.0 gezet en de eind IPv4 adres op 255.255.255.255. Nadat de tests waren uitgevoerd, werd deze regel meteen terug verwijderd. Dit werd op deze manier geregeld, omdat dit niet binnen de afgebakende scope van dit onderzoek lag. 
    \item[Schema en Databasetabellen:]
    In de code werd er gebruikgemaakt van Entity Framework. De databasetabellen werden gedefinieerd via een code-first approach. Dit betekent dat de eigenschappen van de tabellen via de code gezet werden. Voor deze entiteiten werden er configuration-klassen voorzien om verdere instellingen specifiek voor de tabel in te stellen. Vervolgens werd er een migration-bestand aangemaakt die de schema en tabellen van de database kan toepassen op een database. Bij elke aanpassing aan de database entiteiten in de code, werd er een nieuwe migration-bestand aangemaakt en opnieuw toegepast op de database, zodat dit up-to-date was.
\end{description}

\subsubsection*{Azure Blob Storage}

Voor het experiment is er nood aan een Blob Storage om de bronbestanden met meetdata, de masterfiles van de teelten met boompunten en het uiteindelijke resultaat in op te slaan. Dit dient als een centrale opslagplaats die bereikbaar is voor beide omgevingen. Net zoals bij de database werd er gekeken naar wat er gebruikt wordt bij de huidige toepassing in het probleemdomein. Hieruit kwam Azure Blob Storage als oplossing naar boven.  
\\
\\
Aangezien dit in Azure ligt, werd een nieuwe \textit{Storage Account} ingericht en toegevoegd aan de bestaande resource group in de subscription van Agromanager. Welke configuratiestappen er genomen werden, wordt in het volgende onderdeel besproken:

\begin{description}
    \item[Accounttype en redundantie:]
     Er is gekozen voor \textit{StorageV2 (general purpose v2)} als type, omdat dit de standaard is. Om de kosten te drukken, werd de redundantie ingesteld op \textit{Locally-redundant storage (LRS)}.
     \item[Regio:]
     Net als bij de database werd dit ingesteld op \textit{West Europe}. Wederom om netwerkvertragingen te beperken voor de container in Azure en om een stabiel referentiepunt te bieden voor Google Cloud Run.
     \item[Containers:]
     Er zijn 3 opslagcontainers manueel aangemaakt in Azure Portal. 1 voor de masterfiles van de teelten, 1 voor de bronbestanden met meetdata (CSV, Shapefile) en 1 voor het wegschrijven van de resultaten.
     \item[Access Tier:]
     Voor de data in de containers werd gekozen voor \textit{Hot Tier} als \textit{Access Tier}. Dit maakt dat de bestanden geoptimaliseerd zijn om continu geraadpleegd te worden. Dit is noodzakelijk in een testrun die 30 keer uitgevoerd wordt en waarbij de data-ophalingen verweven zitten in geneste lussen. Het zorgt enerzijds voor een hogere opslagkost, maar anderzijds voor een lagere kost per transactie wat bij de benchmarks een kenmerk is.

\subsection{Messaging Infrastructuur}

De dataset wordt asynchroon verwerkt, omdat de gebruiker dan meteen een indicatie krijgt dat de achtergrondverwerking gestart is en hier niet minutenlang op moet wachten. Eerder (\ref{ch:messaging-queue}) werd aangekaart dat bij het testen naar boven kwam dat de schaling via Azure Storage Queue niet werkte zoals het vereist was voor Azure Container Apps. Anderzijds gebeurde de schaling in Google Cloud Run ook niet naar behoren, omdat het standaard geen native ondersteuning biedt voor het schalen op basis van de lengte van een wachtrij. Er diende voor beide omgevingen een alternatieve oplossing gevonden te worden. Bijgevolg zijn Azure Service Bus en Google Cloud Pub/Sub Push ingezet om de orchestratie van de berichtenwachtrij te faciliteren en de bijhorende automatische schaling aan te sturen. Hoe deze toepassingen opgezet zijn, wordt verder in dit hoofdstuk besproken.

\subsubsection*{Google Cloud Pub/Sub}

Om de asynchrone verwerking op een schaalbare manier uit te voeren in Google Cloud Run werd een Pub/Sub-architectuur gebruikt. Dit kan aanschouwd worden als de tegenhanger van Azure Service Bus en werd op de volgende manier ingesteld:

\begin{description}

    Heel cruciaal binnen deze configuratie is de \textit{Acknowledgement Deadline}-instelling. Dit werd verhoogd naar 600 seconden (10 minuten). Aangezien de verwerking 4 tot 6 minuten actief is, voorkomt deze ruime marge dat Google Cloud het bericht als "niet afgeleverd" beschouwt en voortijdig opnieuw aanbiedt (redelivery). Hiermee krijgt de runtime-omgeving de nodige rechten om de status van berichten te beheren. In de instellingen moet je dan wel aanduiden dat de beveiliging via IAM gebeurd.
\end{description}

    Voor elke teelt werden de teeltrijen geometrisch gedefinieerd. Deze actie genereert een masterfile voor elke teelt en bevat alle boompunten met hun respectievelijke locatie.
    \item[Bronbestanden met meetdata:] 
    De ingetekende teelten werden geëxporteerd naar een Shapefile. Deze Shapfiles werden vervolgens geïmporteerd in QGIS. In deze tool werden de boomvlakken omgezet naar boompunten. Tot slot werden er bij de dataset voor Shapefile 7 verschillende Shapefile-bestanden toegevoegd. Dit werd ook gedaan voor de CSV-dataset, maar dan met CSV-bestanden. Elk bestand staat voor een specifieke teelt.

    Dit resulteert erin dat er een iteratie gebeurt per teelt. Per teelt wordt er geloopt over alle bestanden die toegevoegd zijn. Stel dat elke teelt 3000 boompunten gemiddeld heeft.     \begin{equation}
        7 * 7 * 3000 * 3000 = 441.000.000
    \end{equation}
    In dit geval wordt er in de code voor 1 testrun 441.000.000 keer geïtereerd.
\end{description}

Het bekomen van deze dataset werd op een iteratieve manier gedaan. Zo werd er per toevoeging van een teelt en het respectievelijke bronbestand gekeken welke impact dit had op verwerkingstijd van de huidige toepassing in het probleemdomein. Hierbij werd bij het uiteindelijk resultaat gefocust op een verwerkingstijd tussen 4 en 6 minuten. Deze tijdspanne fungeert als \textit{baseline} voor het onderzoek.

        \item SqlConnectionString
        \item AzureWebJobsStorage
        \item AzureStorageAccount
        \item APPLICATIONINSIGHTS\_CONNECTION\_STRING
        \item GCP\_PROJECT\_ID

Bij deze testopstelling is het belangrijk om te benadrukken dat er een beperking naar boven is gekomen voor de POST endpoint die gebruikt wordt door Pub/Sub. Aangezien de verwerkingslogica soms 4,5 minuten duurt, blijft de request naar de endpoint heel de tijd openstaan. Hierdoor wordt er tegen een timeout gelopen. In Pub/Sub en in Google Cloud Run werden de request timeouts verhoogd naar 600 seconden. Echter bleef diezelfde foutmelding plaatsvinden. De volgende zaken werden geprobeerd om het probleem op te lossen:

\begin{itemize}
    \item host.json manipuleren om de timeout te verhogen in de applicatie
    \item fire-and-forget approach in de verwerkingslogica
\end{itemize}

Deze maatregelen brachten geen oplossing. Het doet vermoeden dat de architectuur van deze opstelling niet ideaal is voor een langlopend proces, aangezien de timeouts van de cloudomgeving wel langere verwerkingstijden tolereren. Om te waarborgen dat de geregistreerde benchmarks niet overschreven werden door een retry, werd er geanticipeerd in de code. Er werd gecontroleerd of de geregistreerde benchmark reeds een eindtijd had voor de Queue-verwerking en wanneer dit het geval was, werd het proces meteen afgebroken door een \textit{return}-statement.

\section{Methodologie en Uitvoering}

Dit bespreekt welke benchmarks er uitgevoerd zijn en waarom dit op deze manier gedaan is. We benoemen bij elke testcase op welke prestatiecriteria er gefocust wordt om tot een resultaat te komen en wat de randvoorwaarden zijn. Vervolgens wordt een protocol gedefinieerd voor de uitvoeringen, zodat bij elke uitvoering er een vast stappenplan is om te waarborgen dat resultaten niet beïnvloed worden door variaties in benchmarking. Tot slot wordt aangegeven hoe de betrouwbaarheid van de resultaten verzekerd wordt.

\subsection{Definitie van de Testcases}

\subsubsection{Warm Start}

Deze testcase verzamelt informatie over hoe de prestaties bij beiden omgevingen zijn wanneer de containers warm draaien. Om dit te bekomen worden er 31 requests serieel verstuurd naar Azure Container Apps en vervolgens ook naar Google Cloud Run. Het resultaat van de eerste aanroep naar elke omgeving wordt verwijderd uit de lijst om zo de cold start die mogelijks plaatsvindt te elimineren.

\subsubsection{Cold Start}

Om een beeld te krijgen van hoe efficiënt de platformen omgaan met een cold start wordt er 30 keer een request verstuurd per testomgeving. Hierbij is het per verwerkingsaanvraag een vereiste dat de container inactief is. In Azure Container Apps staat vast wanneer deze service inactief wordt door de \textit{cooling down}-instelling. Bij Google Cloud Run kan dit niet echt ingesteld worden, maar is er wel een manier om dit te forceren door voor elke testrun een nieuwe \textit{revision} te deployen. Aangezien er  gemeten moet worden hoe snel de service op beide platformen een instantie kan opstarten bij een inactieve staat, is het noodzakelijk om andere ruis zoveel mogelijk uit te sluiten. Netwerkvertraging is hier een voorbeeld van. Om dit te beperken worden er per testopstelling 10 requests ´s morgens, 10 ´s middags en 10 ´s avonds uitgevoerd. Dit vangt de variatie op in internetverkeer en datacenterbelasting. Bij deze testcase wordt er enkel gekeken naar de vertraging door container-activatie. Daarom wordt bij de benchmarks gekozen om het bestandstype hetzelfde te houden.

\subsubsection{Schaalbaarheid}

In dit experiment wordt gelogd hoe performant de services schalen bij een \textit{burst-load}. Om dit te bekomen wordt er per platform 10 keer een groep van 5 berichten tegelijk in de wachtrij van de desbetreffende applicatie geplaatst. Hierdoor wordt de HttpTrigger omzeild en wordt er puur gekeken naar het schaalmechanisme van de service. In deze testcase is het bestandstype niet relevant en kan dit vrij gekozen worden. Om consistentie te waarborgen, wordt wel aangeraden om éénzelfde type te kiezen over alle \textit{burst-load}-benchmarks heen.

\subsubsection{Bestandstypes}

Deze testcase volgt hetzelfde proces als de ``Warm Start''-test. 31 runs, waarbij de eerste verwijderd wordt uit de resultaten om een warme start te garanderen. Echter wordt in deze context het proces herhaald voor beide bestandstypes. Deze records worden gebruikt om te bepalen of de datastructuur van de bronbestanden invloed heeft op de rekenkracht of I/O-efficiëntie. De resultaten van de CSV-metingen worden vergeleken met die van de Shapefile-metingen om tot een conclusie te komen.

\subsection{Uitvoeringsprotocol}

Het uitvoeringsprotocol bevordert de reproduceerbaarheid van de experimenten door een strikte volgorde van stappen te definiëren:

\begin{description}
    \item [Pre-check:] Er wordt gecontroleerd of alle queues leeg zijn en of beide services in een inactieve staat zijn (nul instanties).
    \item [Automatisering:] Het Python-script wordt gestart en de testcase wordt uitgevoerd. Tijdens de verwerking van de aanvragen wordt gemonitord hoe alles verloopt en of er geen foutmeldingen of andere inconsistenties zijn.
    \item [Foutafhandeling:] Bij het plaatsvinden van een error, wordt de fout verholpen en worden de benchmarkresultaten gereset voor deze testcase. Vervolgens wordt er terug gestart bij de ``pre-check''-stap.
    \item [Intervalbeheer:] Voor de \textit{cold start}-testcase hanteert het script een verplichte pauze van 120 seconden (voor Azure) of triggert het een \textit{revision update} via de CLI (voor Google) tussen elke run.
    \item [Resultatenverzameling] Na het voltooien van de testruns wordt de ruwe data (start- en eindtijden) gelezen uit SQL Database en aangevuld door extra \textit{metrics} via Kusto-queries (KQL) uit Application Insights.
\end{description}

\subsection{Validatie en Betrouwbaarheid}

Om de betrouwbaarheid van de resultaten te garanderen, zijn de volgende maatregelen genomen:

\begin{description}
    \item [Statistische Relevantie:] Per variabele wordt een steekproefgrootte van $n=30$ gehanteerd om een normaalverdeling te bekomen en de invloed van uitschieters te beperken.
    \item [Isolatie:] Tijdens de runs vinden er geen andere intensieve operaties plaats op de database binnen de Agromanager-omgeving. Dit voorkomt negatieve beïnvloeding op de prestaties van de CPU.
    \item [Synchronisatie:] De \textit{Lock Duration} en \textit{Request Timeouts} op beide platformen zijn ingesteld op 600 seconden om voortijdige onderbrekingen van de $O(n^4)$ berekening te voorkomen.
    \item [Resources:] Op beide omgevingen werden alle instellingen zo gelijk mogelijk ingesteld. Hierbij zijn CPU en geheugen de voornaamste parameters.
\end{description}

In het script wordt de \textit{W3C Trace Context} standaard gebruikt. Voor elke request die het script verstuurt, wordt er een unieke \textit{traceparent} header gegenereerd. Deze unieke ID reist mee door het hele proces: van het Python-script naar de cloudomgevingen en uiteindelijk tot in de SQL-database. Deze ID is cruciaal voor de analyse. Het zorgt ervoor dat de tijd die lokaal gemeten wordt, rechtstreeks gekoppeld kan worden aan de logs in de \textit{BenchmarkRun}-tabel en de gedetailleerde metrics in \textit{Application Insights}. \subsection{Implementatie van de Uitvoeringsprotocollen}

In het script zijn de afspraken uit de methodologie vertaald naar concrete code:

\begin{description}
    \item [Intervalbeheer:] Bij de \textit{cold start}-metingen voor Azure Container Apps werd er een pauze van 120 seconden ingebouwd via de \textit{time.sleep()}-functie. Dit is ruim voldoende om de \textit{cooldown period} van 60 seconden te overbruggen, zodat de container gegarandeerd naar nul is geschaald.
    \item [Geforceerde Cold Starts in GCR:] Omdat Google Cloud Run containers vaak lang "warm" houdt, wordt de \textit{Google Cloud SDK} gebruikt. Na elke run voert het script via een \textit{subprocess} een commando uit naar de CLI van \textit{gcloud} om een nieuwe revisie geforceerd te deployen. Dit zorgt ervoor dat de oude instantie(s) meteen worden afgesloten en de volgende aanvraag een echte cold start is.
    \item [Simulatie van Burst Load:] Om de schaalbaarheid te testen, wordt de \textit{ThreadPoolExecutor} gebruikt. Hiermee worden er parallel 5 berichten in de wachtrij geplaatst. Dit simuleert een piekmoment en laat zien hoe agressief KEDA (bij Azure) en het Pub/Sub-mechanisme (bij Google) nieuwe instanties bijschakelen.
\end{description}

\subsection{Validatie en Database-interactie}

Voordat een testreeks start, doet het script eerst een \textit{pre-check}. Omdat de Azure SQL Database serverless is, wordt eerst een dummy-connectie verstuurd om de database "wakker te maken". Hiermee wordt voorkomen dat de opstarttijd van de database de eerste meting vervuilt. Zodra de testruns klaar zijn, maakt het script via de \textit{pyodbc}-library verbinding met de database. De tijden die daar geregistreerd zijn, worden opgehaald en naast de lokaal gemeten tijden gelegd. Deze dubbele check is een extra zekerheid dat de data klopt voordat er begonnen wordt aan uiteindelijke analyse.
